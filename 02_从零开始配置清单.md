# 从零开始配置清单 Draft

这份清单用于我或同事在一台新电脑上从零配置宏达报销 WorkBuddy 轻量入口。

## A. 准备目录

- [ ] 我先决定这台电脑的 runtime 根目录。

推荐：

`~/Documents/报销管理/hongda_reimbursement_runtime`

- [ ] 我确认不要使用 Downloads、Desktop、聊天缓存或邮件附件缓存作为长期目录。
- [ ] 我建立以下目录：

```text
hongda_reimbursement_runtime/
  inbox/
  output/
  config/
  scripts/
  logs/
  archive/
  approved/
  rejected/
  processing/
```

- [ ] 我确认 WorkBuddy 能读取该目录。
- [ ] 我确认 WorkBuddy 能写入 `output/`。

## B. 准备脚本和配置

- [ ] 我把 runtime 脚本放入 `scripts/`。
- [ ] 我把配置文件放入 `config/`。
- [ ] 我检查至少包含：
  - `config/reimbursement_runtime_config.json`
  - `config/expense_category_mapping.json`
  - `config/expense_subject_allowed_values.json`
  - `config/employee_mapping.json`
  - `config/manual_required_steps.md`
- [ ] 我确认员工映射已按当前使用人更新。
- [ ] 我确认项目映射或项目文件夹命名规则已说明。
- [ ] 我确认模板允许费用科目来自公司原始报销模板，而不是模型自由生成。

## C. 准备 Python 环境

- [ ] 我确认本机能运行 `python3`。
- [ ] 我确认脚本能导入 Excel 依赖，例如 `openpyxl`。
- [ ] 我先运行初始化脚本测试目录权限。
- [ ] 如果 Python 不可用，我先停止，不让 WorkBuddy 继续猜测。

## D. 放入测试发票

- [ ] 我先使用少量非敏感测试文件。
- [ ] 我把文件放入：

`inbox/202606_测试项目_HDC_TEST_001/`

推荐结构：

```text
202606_测试项目_HDC_TEST_001/
  发票/
  附件/
  行程单/
  说明/
```

- [ ] 我确认原始文件不放在准备上传 GitHub 的目录中。

## E. 导入 WorkBuddy Skill

- [ ] 我把 `01_WorkBuddy宏达报销轻量Skill.md` 导入 WorkBuddy。
- [ ] 我确认 Skill 中没有把我的本地路径当成所有同事的固定路径。
- [ ] 我让 WorkBuddy 先询问或确认 runtime 根目录。
- [ ] 我让 WorkBuddy 只做基础队列测试，不做正式出包。

## F. 最小闭环测试

- [ ] T01：Skill 能加载。
- [ ] T02：WorkBuddy 能读取 runtime 根目录。
- [ ] T03：WorkBuddy 能列出 `inbox/` 文件或项目文件夹。
- [ ] T04：WorkBuddy 能调用 Python 初始化脚本。
- [ ] T05：WorkBuddy 能写入 `output/`。
- [ ] T06：WorkBuddy 能生成 Excel 人工确认队列。

预期生成：

- `01_invoice_inventory_draft.xlsx`
- `02_human_review_queue_draft.xlsx`
- `03_missing_materials_checklist_draft.md`
- `04_runtime_execution_report.md`

## G. 真实项目测试

- [ ] 我选择一个真实项目文件夹。
- [ ] 我确认项目文件夹内材料已按项目隔离。
- [ ] 我运行项目文件夹模式。
- [ ] 我确认输出只包含该项目，不混入其他项目。
- [ ] 我打开人工确认队列，检查项目、金额、日期、科目、附件提示。
- [ ] 我确认所有不确定项都有 `Human Review Required` 或人工确认提示。

## H. 失败场景测试

- [ ] 我故意指定一个不存在的项目文件夹，确认 WorkBuddy 会停止并说明错误。
- [ ] 我测试空 `inbox/`，确认 WorkBuddy 不会伪造结果。
- [ ] 我测试没有 PDF 生成能力时，确认 WorkBuddy 能报告限制。
- [ ] 我确认失败时不会移动、删除、改名原始文件。

## I. 推广前确认

- [ ] 我确认同事知道 WorkBuddy 只生成 Draft。
- [ ] 我确认同事知道 `Human Review Required` 不能直接提交。
- [ ] 我确认同事知道最终 OA 提交、打印、贴票、财务判断仍由人工完成。
- [ ] 我确认同事电脑路径、权限、Python、配置都已单独测试。
- [ ] 我确认 GitHub 上传前会完成脱敏。

## J. 通过标准

只有当以下条件全部满足，我才开始让同事试用：

- WorkBuddy 能稳定加载轻量 Skill。
- WorkBuddy 能识别同事自己的 runtime 根目录。
- WorkBuddy 能读取 `inbox/` 并写入 `output/`。
- WorkBuddy 能生成 Excel 和 Markdown 报告。
- WorkBuddy 遇到错误会停止并提示人工处理。
- 原始发票和模板全程未被修改。
